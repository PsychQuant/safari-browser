# Changelog

## Unreleased

### New Features

- **`--mark-tab` / `--mark-tab-persist` flags + `tab is-marked` / `tab unmark` subcommands — Tab ownership marker (opt-in)** — Lets multi-agent and multi-process automation environments publish "I'm working on this tab" via an invisible zero-width Unicode marker on the Safari tab title, queryable by sibling invocations through `safari-browser tab is-marked` (exit 0 = marked, 1 = unmarked, 2 = error). Default OFF — every existing subcommand's behavior is byte-identical when no opt-in flag is passed. Three opt-in signals: `--mark-tab` (ephemeral, wrap then unwrap), `--mark-tab-persist` (wrap and leave for `tab unmark` cleanup), and `SAFARI_BROWSER_MARK_TAB=1` env var (env value `2` or `persist` selects persist mode; flags always win over env). Marker is a hardcoded zero-width-space pair (`U+200B`) at the start and end of the original title — invisible to humans, queryable by code, with no caller-supplied content (security: prevents the marker from becoming a side-channel for agent identity leaking through AX / Spotlight / Stage Manager / screen recording). Title-restore is best-effort: if the page navigates or JS rewrites `document.title` mid-operation, cleanup emits a single `[mark-tab: title changed during operation; original not restored]` stderr line and skips force-restore (avoids Layer-3 oscillation). **v1 scope**: `ClickCommand` is the reference integration; broader command rollout and daemon-protocol-level wrap-once-per-request optimization deferred to v2. The wrap-and-unwrap wrapper at the command-run() level satisfies the spec's "wrap once per request" intent today via a closure-shaped API. 30 new unit tests (`MarkerHelpersTests` 17 + `MarkTabRouterTests` 13) covering marker constants, wrap/unwrap idempotence, env+flag precedence, mutual-exclusion validation. Spec in `openspec/specs/tab-ownership-marker/spec.md`.

- **`safari-browser exec` subcommand — JSON-script batch execution** — Lets agents and CI pipelines run multi-step automation in a single invocation with shared target resolution, variable capture, and minimal conditional flow. Reads a JSON array of step objects (`{cmd, args, var, if, onError}`) from `--script <path>` or stdin. Variable capture: `"var": "name"` binds the step's stdout to `$name`; later steps reference via `$name` substitution (escape literal dollar with `\$`; `\\$` is also valid in JSON). Conditional `if:` expressions support exactly three operators (`contains`, `equals`, `exists`) — no boolean combinators, no parens — keeping host-language complexity outside the binary. Default cap of 1000 steps per invocation (override via `--max-steps`). Default `onError: "abort"` stops at the first failed step; opt-in `"continue"` proceeds. Output is a single JSON array on stdout: `[{"step": N, "status": "ok"|"error"|"skipped", "value": ..., "var": "..."?}, ...]` so consumers can `jq '.[] | select(.status == "error")'` trivially. **v1 scope**: dispatch via subprocess to the same `safari-browser` binary; daemon-shared-connection optimization deferred to v2 (each step still rides the warm daemon at ~50ms when daemon mode is active). Phase 1 commands covered (anything routed through the daemon's Phase 1 list); `screenshot`, `pdf`, and `upload` raise `unsupportedInExec`. 36 unit tests in `ScriptInterpreterTests.swift` covering parser, variable substitution, expression evaluator, max-steps cap, and result-array shape; `make test-exec-script` runs the live-Safari integration test. Spec in `openspec/specs/script-exec/spec.md`.

- **#31 `save-image` subcommand — download DOM element raw resource to disk** — `safari-browser save-image <path> --element <selector>` downloads the raw bytes behind an image / video / audio / picture / inline SVG element. Complementary to `screenshot --element` (#30): that produces a lossy rasterized PNG via AX + CoreGraphics; this produces lossless original bytes via JS resource discovery + HTTP GET. Does **not** require Accessibility permission. Works on `<img>` (reads `currentSrc` falling back to `src`), `<source>` + `<picture>` (resolves to the chosen child), `<video>` / `<audio>` (reads `currentSrc`, or `poster` when `--track poster` is passed), and inline `<svg>` (writes `outerHTML` directly — no HTTP round-trip). Multi-match uses the same fail-closed contract as `#30`: ambiguous selectors throw `elementAmbiguous` listing each candidate's rect + `tag.class#id` + first-60-char text snippet so the user refines the selector or adds `--element-index N` (1-indexed, document order; out-of-range throws `elementIndexOutOfRange`). Five new error cases: `elementHasNoSrc`, `unsupportedElement`, `downloadFailed`, `downloadSizeCapExceeded`, `unsupportedURLScheme` (blocks `javascript:` / `file://` / `data:` URIs at validate time). A download hard cap (`--max-bytes`, default 200 MB) surfaces `downloadSizeCapExceeded` before allocating the full buffer. 34 unit tests + deterministic fixture `Tests/Fixtures/save-image-test.html`. Spec in `openspec/specs/save-image-subcommand/spec.md`. Closes #31.

- **#34 Precise URL matching: `--url-exact`, `--url-endswith`, `--url-regex`** — Three new targeting flags that complement `--url <substring>` when hierarchical URL schemes produce prefix-substring ambiguity. `--url-exact "<url>"` requires byte-identical match (no trailing-slash / host-case / percent-encoding normalization — caller must pass the URL exactly as Safari stores it, per Non-Goals). `--url-endswith "<suffix>"` is the primary escape hatch for parent-vs-child URL pairs (`.../lesson/X` vs `.../lesson/X/video/Y/auth/Z/play`): pass a unique suffix like `d43e` or `/play` to single out the deepest-matching tab. `--url-regex "<pattern>"` uses `NSRegularExpression` default options (case-sensitive, unanchored — add `^...$` for full-string match); invalid patterns fail fast at CLI validate time with the compiler's error description. All four URL-matching flags (`--url`, `--url-exact`, `--url-endswith`, `--url-regex`) are mutually exclusive with each other and with `--window / --tab / --document`; empty `--url-endswith ""` is rejected with a clear validation error. Accessible to every subcommand that takes `TargetOptions` (30+ commands). Closes #34.

### Bug Fixes

- **#33 `--first-match` now actually works on read-path commands** — `--first-match` was parsed into `TargetOptions.firstMatch` and the `SafariBridge.resolveNativeTarget(firstMatch:warnWriter:)` resolver implemented the fallback correctly, but `resolveToAppleScript` (the read-path dispatcher used by `js`, `get url / title / text / source`, `snapshot`, `storage`, `wait`, `click`, `fill`, `upload --js`, and every other `target.resolve()` → `SafariBridge.doJavaScript*` pipeline) hardcoded `firstMatch: false` when calling the resolver. Every read-path command emitted the documented `"Pass --first-match to accept the first match"` error **even when `--first-match` was already on the command line** — the suggestion was a dead loop. Fix plumbs `firstMatch` + `warnWriter` default parameters through `resolveToAppleScript` and every read-path bridge entry point (`doJavaScript`, `doJavaScriptLarge`, `getCurrentURL`, `getCurrentTitle`, `getCurrentText`, `getCurrentSource`, `openURL`), and adds `TargetOptions.resolveWithFirstMatch()` + `TargetOptions.stderrWarnWriter` helpers so every 30+ command that composes `TargetOptions` forwards the flag uniformly. Also adds `SafariBridge.resolveToConcreteTarget` which commands making multiple bridge calls (notably `JSCommand`'s store / read-length / read-result / delete sequence) use to collapse `.urlMatch` → `.windowTab` once at the command boundary — this guarantees the `--first-match` warning fires **at most once per invocation** and eliminates a tab-list-race between chunked reads. Closes #33. Closes #35 as invalid (focus-existing was already implemented in commit `68a64bc` / tab-targeting-v2; reporter's real pain was the #34 hierarchical-URL substring ambiguity, now solved by `--url-endswith`).

### Breaking Changes

- **#28 tab-targeting-v2: `open <url>` default changed from "navigate front tab" to "focus-existing → else new-tab"** — When a Safari tab already contains the requested URL (exact match), `safari-browser open <url>` now focuses that tab via the spatial interaction gradient (Layer 1 noop / Layer 2 tab-switch / Layer 3 window raise / Layer 4 cross-Space new-tab) instead of navigating the front window's current tab via `window.location.href=...`. Scripts that relied on the navigate-front-tab behavior must migrate to `safari-browser open --replace-tab <url>`, which preserves the legacy semantics. Motivated by the Human Emulation principle: users who press Cmd+L and type a URL already open expect the browser to focus the existing tab, not overwrite the current one.
- **#28 tab-targeting-v2: `.urlContains` multi-match unified fail-closed across all subcommands** — Previously, JS-path subcommands (`js`, `open`, `get`, `storage`, `wait`, `snapshot`) went through `(first document whose URL contains "...")` AppleScript which silently picked the first match; Native-path subcommands (`upload --native`, `close`) fail-closed with `ambiguousWindowMatch`. After this change every `--url <substring>` that matches >1 tab throws `ambiguousWindowMatch` regardless of subcommand. Scripts that depended on silent first-match must explicitly pass `--first-match` (which also emits a stderr warning listing all candidates) or use the new `--window N --tab-in-window M` structured flag to target a specific tab.
- **#28 tab-targeting-v2: `--tab N` deprecated (still functional, removed in v3.0)** — `--tab N` was an alias for `--document N` (global document index). In the new tab-bar model "tab" should mean "Nth tab within a window", so the alias is misleading. Every `--tab N` invocation now emits a stderr deprecation warning advising migration to `--document N` (to keep current semantics) or `--window M --tab-in-window N` (to address a specific tab within a window). The flag will be removed in the next major version.
- **#28 tab-targeting-v2: `safari-browser documents` output format changed to tab-level enumeration** — Previously output one line per window (each window's front tab only). Now outputs one line per tab across all windows, with columns `[global_index] <current-marker> w<window>.t<tab_in_window>  <url> — <title>` and `--json` schema adds `window`, `tab_in_window`, `is_current` fields. Text parsers that counted document lines as "number of Safari windows" will see higher counts; they must either group by the `w{N}` prefix or switch to `--json`. The previous canonical mapping "documents N = upload target N" is preserved because `--document N` continues to address the Nth tab in spatial enumeration order.

### New Features

- **#30 screenshot `--element <selector>` + `--element-index N`: crop to a specific DOM element** — New option on `safari-browser screenshot` that captures only the bounding rectangle of a CSS-selector-matched element. Eliminates the external-tool step in batch card/component/chart capture workflows (teaching screenshots, `agent-browser` visual diff, prompt-library curation). Coordinate pipeline: `getBoundingClientRect` (viewport-relative points) + `AXWebArea origin − window origin` (from #29's `getAXWebAreaBounds`) → window-relative rect → pixel rect via `ImageCropping.pixelRect`/`scale` → `cropPNG`. **Multi-match disambiguation**: fail-closed default; `--content-only`'s alternative remains "re-run without `--content-only`"; unknown flags get a generic "run without <flag>" fallback. New abstractions: `SafariBridge.getElementBoundsInViewport(selector:target:elementIndex:)` (JSON-encodes selector to survive quote/backslash/Unicode injection), `SafariBridge.parseElementBoundsResponse(_:selector:)` (pure function, unit-testable without live Safari), `ElementBoundsResult` and `ElementMatch` structs. Spec delta (5 ADDED Requirements, 12 scenarios) merged into `openspec/specs/screenshot/spec.md` via `spectra-archive` (spec grew from 9 → 14 requirements). Pairs with #31 (`save-image` subcommand for original-resource download) as the complementary lossy-vs-lossless element-capture option.
- **#29 screenshot `--content-only`: crop Safari chrome, keep only web content area** — New flag on `safari-browser screenshot` that outputs a PNG containing only the AXWebArea region (URL bar, tab bar, toolbar are cropped out). Eliminates the batch-workflow pitfall where users fell back to `screencapture -x -R x,y,w,h` and lost the `-l <windowID>` z-order guarantee (any window covering Safari would leak into the screenshot). New pure utility `Utilities/ImageCropping.swift` exposes `cropPNG / scale / pixelRect / isNoOpCrop` as Safari-free functions for unit testability. 16 new unit tests (11 ImageCroppingTests + 2 ErrorsTests + 3 boundary cases); integration verified on live Safari (3736×2304 → 3736×2064, 120 pt chrome removed) and `--full --content-only` stable across 3 consecutive runs. Spec delta (5 ADDED Requirements) merged into `openspec/specs/screenshot/spec.md` via `spectra-archive`. Follow-up work tracked as #30 (`--element <selector>` for element-scoped crop, reuses this infra) and #31 (`save-image` subcommand for original resource download).
- **#28 tab-targeting-v2: `--window M --tab-in-window N` composite flag** — Structured addressing for duplicate-URL tabs. `--tab-in-window` requires `--window` (standalone usage is a validation error). The pair resolves to the Nth tab within Safari's Mth window, giving the previously-unreachable "second plaud tab in window 1" case a direct target syntax. Accessible to every subcommand that takes `TargetOptions` (30+ commands).
- **#28 tab-targeting-v2: `--first-match` opt-in for `--url` ambiguous matches** — Restores silent-first-match behavior when explicitly requested. Unlike the pre-change implicit behavior, `--first-match` emits a stderr warning enumerating all matches and indicating which was selected, so debugging a silently-wrong-target scenario is possible. Deterministic selection: lower window index → lower tab-in-window index wins.
- **#28 tab-targeting-v2: `--replace-tab` opt-in for `open`** — Opt-out of the new focus-existing default. Scripts that need "navigate the front tab to this URL even if another tab has it" behavior can add `--replace-tab`. Mutually exclusive with `--new-tab` / `--new-window`.
- **#28 tab-targeting-v2: Spatial interaction gradient for `open --focus-existing`** — 4-layer policy that decides how to reveal an existing tab based on its spatial relationship to the current frontmost window: noop (same tab), tab-switch (same window), window raise (same macOS Space), or new-tab-in-current-Space (cross-Space). The cross-Space branch intentionally does NOT raise windows from other Spaces — prevents automation from dragging the user's Mission Control layout across Space boundaries. Space detection uses private CGS SPI (`CGSGetActiveSpace` / `CGSGetWindowWorkspace`); when unavailable, policy falls back to Layer 3 (conservative). Classification in the `non-interference` spec: Layers 1 & 4 non-interfering, Layer 2 passively interfering (no warning), Layer 3 passively interfering (stderr warning).
- **#28 tab-targeting-v2: New `human-emulation` design principle** — Cross-cutting principle co-equal with `non-interference`. Four derived rules: tab-bar is ground truth (not the `documents` collection), ambiguity fails-closed (never silent first-match), known URLs focus instead of reopen, spatial interaction is gradient-based. Full spec at `openspec/specs/human-emulation/spec.md` (generated on archive). CLAUDE.md now documents both principles side-by-side.

Users with a typo'd `--window 99` saw a misleading "keyboard is being controlled" warning for a run that failed immediately without touching any keys. - **#23 verify R2: regression guard for `--window N` → `.windowIndex(N)` mapping** — R2 devil's-advocate flagged that the parsing tests covered `TargetOptions.resolve()` and `WindowOnlyTargetOptions.window` individually but NOT the per-command `--window N` → `.windowIndex(N)` mapping used by ScreenshotCommand's `docTarget` and UploadCommand's `jsTarget`. A regression flipping back to `.documentIndex($0)` (the original R0 bug) would have passed all tests silently. The mapping is now centralized on `SafariBridge.TargetDocument.forWindow(_:)` (static helper) and `WindowOnlyTargetOptions.resolveAsTargetDocument()` (struct pass-through). Three new unit tests lock the invariant: `testTargetDocument_forWindow_mapsIntToWindowIndex`, `testTargetDocument_forWindow_nilMapsToFrontWindow`, `testWindowOnlyTargetOptions_resolveAsTargetDocument_matchesForWindow`.
- **#23 verify R1: `screenshot --window N` / `upload --native --window N` captured the wrong document** — `ScreenshotCommand` and `UploadCommand` both mapped `--window N` to `SafariBridge.TargetDocument.documentIndex(N)`, which resolves to Safari's global `document N` — the Nth document in the recency-ordered document collection, **not** `window N`'s current tab. Two windows, two tabs each, and `screenshot --window 2 --full` would read dimensions from the wrong document, resize window 2 to those wrong dimensions, and leave another tab's scroll position mutated. `upload --window 2 --native` was worse: the JS `el.click()` could land on a file input in a completely different window, then raise window 2 to the front and send keystrokes into a sheet that was attached elsewhere. Both paths now use `.windowIndex(N)` → `document of window N`. Discovered via 6-AI verify round 1 (devil's-advocate + logic + codex, independent).
- **#23 verify R1: `getWindowID(window: N)` could silently target the wrong CG window** — The previous name-matching resolver failed in three ways: (1) two Safari windows with identical titles (e.g., two Gmail "Inbox" tabs) → CG scan picked the first z-order match, not window N; (2) `hasPrefix` fuzzy match → `"Example"` matched `"Example — Extra"`, picking window 1 when the user asked for window 2; (3) name-match miss fallback returned the first Safari window with `height > 100`, completely ignoring `--window N`. Rewrote as two private helpers: `getWindowIDByBounds(windowIndex:)` (used when `--window` is set) asks Safari for `bounds of window N`, then scans CG windows for a matching rect within ±2 pts — bounds are globally unique per window and unambiguous. `getFrontWindowID()` (used when `--window` is absent) keeps the legacy name-based flow but drops the `hasPrefix` fuzzy match for exact equality. Missed matches in the targeted path error out with `noSafariWindow` instead of silently falling back to "first Safari window with height > 100".
- **#23 verify R1: `wait --url <pattern>` silently broke pre-#23 scripts** — After the `--url` → `--for-url` rename, the old syntax parsed `--url` as a targeting flag and failed `validate()` with a cryptic `"Provide milliseconds, --for-url, or --js"` error — no hint that `--url` had been repurposed. `WaitCommand.validate()` now detects `target.url != nil && forUrl == nil && js == nil && milliseconds == nil` and throws ``wait --url <pattern> was renamed to wait --for-url <pattern> in #23 — --url is now a global targeting flag. Retry as `safari-browser wait --for-url "<pattern>"` (see CHANGELOG).`` The `testWaitCommand_oldUrlFlagRejectedWithRenameHint` test asserts the rename-hint is present; a new `testWaitCommand_missingConditionWithNonUrlTarget` test confirms `--document N` alone still falls through to the generic error.
- **#23 verify R1: `close --window 99` / `pdf --window 99` surfaced raw AppleScript errors** — `SafariBridge.closeCurrentTab(window:)` called `runAppleScript` directly; `PdfCommand.raisePrelude` went through `runShell` — both bypassed the `runTargetedAppleScript` wrapper that all other targeted commands use for error translation. Users saw raw `-1719` / `-1728` AppleScript spew instead of the consistent `documentNotFound` error listing every available document. `closeCurrentTab` now routes through `runTargetedAppleScript` with `target: .windowIndex(window)` when a window is supplied; `PdfCommand` and `UploadCommand --native` both preflight via `getCurrentURL(target: .windowIndex(window))` BEFORE touching System Events, so invalid window indices error out before any keyboard-takeover warning is printed.
- **#23 verify R1: `openspec/specs/wait/spec.md` and `PLAN.md:116` still referenced `wait --url`** — Phase 5 docs missed the live Spectra spec and the repo-root `PLAN.md`. Both now reference `--for-url`; the spec gained a new "Legacy `--url` syntax gets rename hint" scenario covering the UX fix above.
- **#19: Wall-clock timeout for osascript / shell subprocesses** — `runShell` and `runAppleScript` used `process.waitUntilExit()` with no timeout, so a stuck osascript (blocked on Safari's Apple Event dispatcher or unresponsive System Events) would hang the whole CLI until `kill -9`. New `runProcessWithTimeout` helper wraps `Process` with a `Task.detached` watchdog that sends SIGTERM, waits 1s, then SIGKILLs; surfaces `SafariBrowserError.processTimedOut(command:seconds:)` on timeout. Default 30s; `upload` uses 60s to accommodate its internal `maxWait to 10` waits.
- **#19 F1 / R2-F1': Bound `--timeout` to a safe finite range** — The first follow-up only rejected `NaN`, `±infinity`, and non-positive values, but `Double.greatestFiniteMagnitude` (and any value like `1e300`) slipped through and still trapped inside `UInt64(timeout * 1e9)`. The guard now enforces `0.001 ≤ timeout ≤ 86_400` in both `UploadCommand.validate()` and `runProcessWithTimeout`, so no reachable input can overflow the nanosecond conversion or round to zero nanoseconds. `SafariBrowserError.invalidTimeout(Double)` surfaces the violation with a clear message.
- **#19 F2 / R2-F2': Double-check `terminationStatus` when reporting a timeout** — The first follow-up latched `didTimeout` before `process.terminate()`, which opened a μs-wide race: if the child exited naturally between the watchdog's `isRunning` check and the SIGTERM, `processTimedOut` was raised despite `terminationStatus == 0`. The main thread now requires both `didTimeout.value == true` *and* `terminationStatus != 0` before classifying a run as a timeout; clean natural exits fall through to the normal success path.
- **#19 F2: Distinguish watchdog kill from external signals** — Previously any `.uncaughtSignal` termination (Ctrl+C propagation, OOM killer, osascript crash) was misreported as `processTimedOut`. The watchdog now flips a `TimeoutFlag` (NSLock-backed) before calling `terminate()`, and the main path only raises `processTimedOut` when that flag is set. Other signal sources fall through to `appleScriptFailed`.
- **#19 F5/F8: Error-message polish** — `processTimedOut` uses `ceil` so sub-second timeouts don't render as "0 seconds", and now includes a troubleshooting hint pointing at Console.app for System Events / Apple Event dispatcher issues.
- **#22: E2E tests are opt-in via `RUN_E2E=1`** — Previously plain `swift test` ran the `E2ETests` suite, whose `setUp` activates Safari and navigates its front window to a fixture page, stealing focus from whatever the user was doing. The suite is now skipped by default and only runs when `RUN_E2E=1` is set. `SKIP_E2E` remains as a redundant opt-out. Brings the test suite in line with the non-interference principle in `openspec/specs/non-interference/spec.md`.
- **#20: Probe System Events before sending keystrokes** — `upload --native` and the PDF export dialog now run a 2-second `probeSystemEvents()` (wrapped inside `runShell(timeout:)` from #19) before touching the keyboard. If the probe is slow (>500 ms) a `⏳ Waiting for System Events...` line is printed to stderr so users never see a silent hang. If the probe fails the bridge prints a loud warning — including the fact that a restart will interrupt other System Events automation (Keyboard Maestro, Alfred, Shortcuts, etc.) — then attempts a best-effort `restartSystemEvents()` (`killall "System Events"` + 500 ms pause for launchd to reap the PID + re-probe; launchd relaunches the process on the subsequent Apple Event). Any remaining failure surfaces as `SafariBrowserError.systemEventsNotResponding(underlying:)`, whose description calls out the user-visible commands (`upload --native`, `pdf`), the paste-able recovery command, and the non-interference warning. `probeSystemEvents(script:timeout:)` accepts an optional `executable:` parameter so tests can exercise the `/nonexistent` binary path. `SystemEventsProbeTests` now `XCTSkipUnless` `/usr/bin/osascript` is executable so sandboxed CI runners don't block on unavailable tools.

### Features
- **#26: Native-path URL resolution for window-only primitives — `upload --native --url plaud` finally works** — Closes the #23 R5 gap that was exposed by #24: when the 10 MB `--js` hard cap pushed all large-file uploads onto the `--native` path, the `--native` path's rejection of `--url / --tab / --document` orphaned every multi-window large-file workflow. Users resorted to shell pipelines like `WIN=$(safari-browser documents | grep -n plaud | cut -d: -f1); safari-browser upload --native --window $WIN ...` — fragile (output-format race), non-deterministic (window re-order between query and execute), and hostile to AI agents. #26 pushes the URL→window resolver down to the Swift layer as `SafariBridge.resolveNativeTarget(from: TargetDocument) async throws -> ResolvedWindowTarget`, with a pure `pickNativeTarget(_:in:)` core that's fully unit-tested without live Safari. A single AppleScript roundtrip (`listAllWindows` with GS/RS-separated output) enumerates every window's tabs with current-tab flag; `pickNativeTarget` short-circuits trivial cases (`.frontWindow`, `.windowIndex(n)`) without enumeration. Multi-match `--url plaud` fails closed with a new `SafariBrowserError.ambiguousWindowMatch(pattern:matches:)` that lists every matching window index + URL — deterministic automation behavior prevails over silent first-match convenience. When the target document is in a background tab of its owning window, native-path commands (`upload --native`, `pdf`, `close`) call `performTabSwitchIfNeeded` to briefly switch tabs before keystroke dispatch; the tab switch is classified as a passively interfering side effect transitively authorized by `--native` / `--allow-hid` (non-interference spec delta). Screenshot deliberately does NOT tab-switch — it observes without interfering, and the AX path continues to read background-window bounds via `_AXUIElementGetWindow` without raising. New error case `ambiguousWindowMatch(pattern:matches:)` joins the existing `documentNotFound` error family. 30+ new tests: 18 in `WindowIndexResolverTests` (pure resolver + parser unit tests), 5 in `ErrorsTests` (new error description), plus per-command target-wiring tests for the 4 migrated commands. 121 → 155 total tests. Closes #26. Downstream consumer `che-local-plugins plaud-transcriber` can now restore `safari-browser upload --native --url plaud ...` syntax without the manual-switch-tab step that was added in v1.8.2.
- **#23: Deferred commands wired through multi-document targeting** — Completes the tech-debt backlog from the `multi-document-targeting` change (Phase 8.4 deferrals). `storage local get/set/remove/clear`, `storage session get/set/remove/clear`, `snapshot` (both interactive and `--page`), `wait --for-url / --js`, and `upload --js` now accept the full `TargetOptions` (`--url`, `--window`, `--tab`, `--document`) and route the underlying `doJavaScript` / `getCurrentURL` calls to the requested document. Multi-window users can finally run `safari-browser storage local get token --url plaud` and get Plaud's per-origin token instead of silently reading from `document 1` (the Devil's Advocate Round 1 finding against #17). `close`, `screenshot`, `pdf`, and `upload --native` expose a new `WindowOnlyTargetOptions` (`--window <n>` only) — their underlying primitives (`close current tab of window N`, CG window ID capture, System Events keystrokes) are inherently window-scoped and parse-time reject `--url` / `--tab` / `--document`. `UploadCommand` also gains split-path validation: `--js` accepts any target, `--native` / `--allow-hid` only `--window`, and the smart default routes to `--js` automatically when the user supplies a document-level target. Internally `SafariBridge.closeCurrentTab(window:)` and `SafariBridge.getWindowID(window:)` accept an optional window index (nil = legacy front-window behavior). `ScreenshotCommand --full` threads the same window index through its dimensions read, bounds save/restore, and CG capture so every moving part stays in sync; `PdfCommand` and `UploadCommand --native` raise the target window via `set index of window N to 1` before activating Safari so keystrokes land on the correct window.
- **#19: `upload --timeout <seconds>`** — Override the native file dialog subprocess timeout (default 60s) for slow machines or large directories. Help text now explains why the default is 60s (three stacked `maxWait to 10` loops inside the combined AppleScript).
- **#17 / #18 / #21: Multi-document targeting (in progress via Spectra `multi-document-targeting` change)** — Introduces a new `SafariBridge.TargetDocument` enum with four cases (`frontWindow`, `windowIndex(Int)`, `urlContains(String)`, `documentIndex(Int)`) and a `resolveDocumentReference(_:)` helper that generates AppleScript document references safe to interpolate into any `tell application "Safari"` block. A new global `TargetOptions` (`@OptionGroup`) plumbs `--url <pattern>` / `--window <n>` / `--tab <n>` / `--document <n>` flags through 25+ subcommands — `get url/title/text/source/html/value/attr/count/box`, `open`, `js`, `click`, `fill`, `type`, `select`, `hover`, `scroll`, `press`, `focus`, `dblclick`, `drag`, `scroll-into-view`, `find`, `check`, `uncheck`, `tabs`, `tab`, `is (visible/exists/enabled/checked)`, `back`, `forward`, `reload`, `highlight`, `cookies (get/set/clear)`, `console`, `errors`, `set media`, and `mouse (move/down/up/wheel)`. The flags are mutually exclusive; `tabs`/`tab`/`open --new-tab`/`open --new-window` reject document-level flags because those commands are window-scoped. Read-only page-info getters (`URL`, `name`, `text`, `source`) and `doJavaScript` now use a document-scoped AppleScript reference (`document N` or `first document whose URL contains …`) instead of `current tab of front window`, so queries bypass any modal file dialog sheet blocking the front window (#21). Single-window usage is byte-for-byte unchanged — `.frontWindow` resolves to `document 1`, which equals `current tab of front window` in that setup. Docs updates and full multi-window verification are deferred to later phases of the same change (see `openspec/changes/multi-document-targeting/tasks.md`).
- **#17 / #18: `safari-browser documents` subcommand** — New discovery command that lists every Safari document's 1-based index, URL, and title in document-collection order — the exact index accepted by `--document N`. Supports `--json` for scripted use. Empty Safari (no documents) is a clean exit-0 with no output (text) or `[]` (JSON).
